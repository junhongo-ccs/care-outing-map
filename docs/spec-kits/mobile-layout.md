# スマホレイアウト Spec Kit

Care Outing Map のスマホ版レイアウト設計メモです。いきなり実装して壊さないために、画面構成、操作フロー、UI部品、実装順、確認観点をまとめます。

## 目的

スマホでは、地図を主役にします。

PC 版は左側パネルを常時表示していますが、スマホでは画面幅が限られるため、検索・ケアモード・候補リストを常時出すと地図操作がしづらくなります。

スマホ版では次を優先します。

- 地図を広く触れる
- 目的地入力から `flyTo` できる
- 候補リストは必要な時だけ開ける
- ピンや候補を選んだら、詳細は下からカードで確認できる
- PC 版の構造を大きく壊さず、レスポンシブ CSS と小さな state 追加で段階的に対応する

ただし、候補選択やピンタップの挙動は JavaScript 側にも分岐が必要です。CSS だけで完結するスマホ対応ではありません。

## 基本構成

スマホでは、次の3層構成にします。

1. 全画面地図
2. ハンバーガーから開く横スライドメニュー
3. 下から出るトイレ詳細カード

## 画面幅の切り替え

まずは `max-width: 768px` をスマホレイアウトの基準にします。

- `769px` 以上: 既存の PC レイアウト
- `768px` 以下: スマホレイアウト

必要に応じて、後で `480px` 以下の細かい調整を追加します。

CSS のメディアクエリだけでなく、JS 側でも同じブレークポイントを参照できるようにします。

候補:

- `window.matchMedia("(max-width: 768px)")`
- `useIsMobileLayout()` のような小さな hook

この判定は、候補カードや地図ピンを1回タップした時に詳細カードを開くかどうかの分岐で使います。

## 操作フロー

### 行き先検索

1. ユーザーがハンバーガーをタップする
2. 左からメニューがスライドして開く
3. 行き先を入力する
4. `検索` をタップする
5. 検索成功時、地図が目的地へ移動する
6. メニューを閉じ、地図を広く見せる

検索後は、候補リストが更新されたことがわかるように、必要なら短いフィードバックを出します。

例:

- `奥沢 1.5km圏`
- `候補 16件`

### ケアモード変更

1. ハンバーガーを開く
2. ケアモードを選ぶ
3. 候補リストとピンが更新される

ケアモード変更後に自動でメニューを閉じるかは要検討です。

初期案:

- ケアモード変更ではメニューを閉じない
- 候補カードを選んだ時にメニューを閉じる

理由:

ケアモード変更後に候補の並びを見比べたい可能性があるため。

### 候補選択

1. ハンバーガーメニュー内の候補カードをタップする
2. メニューが閉じる
3. 地図が候補トイレへ寄る
4. 選択中ピンが強調される
5. 下から詳細カードを表示する

PC 版では「1回目で選択、2回目で詳しく見る」の挙動ですが、スマホでは候補をタップした時点で詳細カードを出すほうが自然です。

### ピン選択

1. 地図上のピンをタップする
2. 地図がそのピンを選択状態にする
3. 下から詳細カードを表示する

スマホユーザーは地図をピンチ・スワイプして操作することに慣れているため、ピンから直接詳細カードを出す流れを優先します。

### ハンバーガー操作と詳細 bottom sheet の関係

詳細 bottom sheet を表示したまま、ユーザーがハンバーガーをタップして drawer を開くケースがあります（例: 詳細を見た後に別の場所を検索し直したい）。

このとき、ハンバーガーのタップは詳細 bottom sheet から見た「外タップ」として扱い、drawer を開くと同時に bottom sheet を閉じます。

理由:

`.modal-backdrop` は `top-bar` の高さぶん上を空けているため（`inset: 56px 0 0` など）、ハンバーガーボタンは backdrop の外側にあります。backdrop の「overlay タップで閉じる」だけに頼ると、ハンバーガーのタップはそこに含まれず、bottom sheet が開いたまま drawer が前面に重なって表示が崩れます。

対応:

ハンバーガーの `onClick` で `setIsMenuOpen(true)` と同時に `setModalToilet(null)` を呼び、bottom sheet を閉じます。z-index の調整では解決しない（bottom sheet と drawer が同時に開いた状態自体を作らないほうが安全）ため、状態側で防ぎます。

## UI 部品ごとの仕様

### `top-bar`

PC 版:

- ブランド
- 現在エリア
- ケアモード

スマホ版:

- 左にハンバーガーボタン
- 中央または左寄せにブランド名
- サブテキストとして現在エリア
- ケアモードは `side-panel` drawer 内へ移動

スマホ版では、上部バーを高さ 56px 程度に抑えます。

注意:

PC 版の `top-bar` は `height: 60px` です。詳細モーダル背景の `.modal-backdrop` は上部バー高さをハードコードしており（PC: `inset: 60px 0 0` / スマホ: `inset: 56px 0 0`）、`top-bar` 高さを変える場合は `.modal-backdrop` の上方向 inset も同じ値へ連動させます。

ハンバーガーボタンの `onClick` では、drawer を開く（`setIsMenuOpen(true)`）と同時に詳細 bottom sheet も閉じます（`setModalToilet(null)`）。理由は「ハンバーガー操作と詳細 bottom sheet の関係」を参照してください。

### `side-panel`

PC 版:

- 左側に常時表示

スマホ版:

- 左からスライドする drawer
- 幅は画面の `min(88vw, 360px)` 程度
- 背面に半透明 overlay を置く
- `×` または overlay タップで閉じる

overlay の重なり順:

- 地図より前面
- drawer より背面
- 詳細 bottom sheet より背面

現在の主な `z-index` は、地図が `0`、`top-bar` / `side-panel` が `20`、モーダルが `30` です。drawer overlay は `z-index: 21`、drawer は `z-index: 22`、詳細 bottom sheet は `z-index: 30` 以上を目安にします。

含めるもの:

- 行き先検索
- ケアモード
- 候補トイレ一覧

アクセシビリティ上の注意:

drawer は CSS の `transform: translateX(-100%)` で画面外に出しているだけで、閉じている間も DOM 上には常駐しています。そのままだとキーボード操作やスクリーンリーダーから、閉じた drawer 内の検索 input やケアモードボタン、候補カードに到達できてしまいます。

スマホ幅かつ `isMenuOpen === false` のときは、`aside.side-panel` に `aria-hidden` と `inert` を付与してフォーカス・読み上げ対象から外します。PC 幅では常時表示なので、この制御は入れません（`isMobileLayout && !isMenuOpen` の時だけ有効にする）。

`inert` は `@types/react` 18系の型定義に含まれないため、`{...({ inert: ... } as Record<string, string | undefined>)}` のようにスプレッドでキャストして渡します。

### `care-mode-control`

PC 版:

- top bar 内の segmented control

スマホ版:

- drawer 内の上部に配置
- 3つのボタンを横並び
- 幅が足りない場合はテキストを短くする

表示例:

- `車椅子`
- `乳幼児`
- `高齢者`

実装上の注意:

現在の DOM では、`care-mode-control` は `<header class="top-bar">` の子で、`side-panel` はその兄弟要素です。CSS の `order` や `display` だけでは別の親コンテナへ移動できません。

スマホ版で drawer 内へ配置するには、次のどちらかが必要です。

- `CareModeControl` コンポーネントとして切り出し、PC 用とスマホ用の配置先で使う
- JSX 側で PC 用とスマホ用を条件レンダリングする

状態は同じ `mode` / `setMode` を共有します。

### `results-list`

PC 版:

- 左パネル内で常時表示

スマホ版:

- drawer 内で縦スクロール
- カードの高さは PC より少し大きめでもよい
- タップしやすいように最低高さ 56px 以上を確保

候補カードをタップしたら:

- drawer を閉じる
- 地図を `flyTo`
- 詳細 bottom sheet を開く

### `ToiletModal`

PC 版:

- 中央モーダル

スマホ版:

- bottom sheet
- 下からスライドして出る
- 幅 100%
- 最大高さは `75vh` 程度
- 写真がある場合は横幅いっぱいで表示
- 閉じる操作は `×` ボタン、overlay タップ、または下方向スワイプ風の UI を検討

初期実装では、スワイプ操作は入れなくてよいです。まずはボタンと overlay で閉じられれば十分です。

実装上の注意:

現状の `.toilet-modal` には `width: min(440px, calc(100vw - 720px))` が指定されています。スマホ幅では `calc(100vw - 720px)` が負値になり得ます。

スマホの bottom sheet 化では、メディアクエリ内でこの PC 用の幅計算を必ず上書きします。

例:

- `width: 100%`
- `max-height: 75vh`
- `border-radius: 18px 18px 0 0`
- `bottom: 0`

実装上の注意（追加）:

base の `.toilet-modal` は `overflow: visible` です。`max-height: 75vh` だけをスマホ用に上書きすると、写真や理由タグが増えたときにカードの中身が画面外へはみ出します。スマホのメディアクエリ内で `overflow-y: auto` も明示します。

### `MapView`

スマホ版でも地図は全画面表示を維持します。

注意点:

- top bar に隠れないようにする
- bottom sheet 表示時、選択ピンが完全に隠れないようにカメラオフセットを調整する
- drawer 表示中は、地図操作を overlay で止める

実装上の注意（追加）:

ピンの `click` / `gmp-click` ハンドラは `handleMapPinClick` を経由し、内部で `isMobileLayout` を参照して1タップ挙動を分岐します。マーカーを再生成する `useEffect` の依存配列に `isMobileLayout` が入っていないと、ウィンドウ幅が 768px 境界をまたいでも既存マーカーが古いモバイル判定のクロージャを掴んだままになります。マーカー描画 `useEffect` の依存配列は `[isMobileLayout, mode, onOpen, onSelect, selected, toilets]` にします。

## カラー方針

Material Design の整理された状態表現と、Apple 的な軽い surface 表現を組み合わせます。

中心色:

- Primary: `#198882`
- Bright primary: `#20b2aa`

UI で使う役割:

- 選択中テキスト: `#198882`
- ボタン枠: `#198882`
- フォーカスリング: `rgba(25, 136, 130, 0.16)`
- 選択中カード枠: `rgba(25, 136, 130, 0.65)`
- ブランド背景: `#20b2aa`
- パネル背景: 半透明白
- 地図: 主役なので UI 色を乗せすぎない

地図ピンのスコア色は、現状では既存仕様を維持します。

実装時の指摘（修正済み）:

`care-mode-options` の hover / focus-visible が `rgba(0, 51, 255, 0.08)` という Apple Blue 系の色のまま残っていました。この spec kit のトーン統一（teal系）から漏れていたため、`rgba(25, 136, 130, 0.08)` / focus リングの `rgba(25, 136, 130, 0.16)` に揃えています。新しく状態色を追加するときは、既存の `--blue`（地図ピン用）と `--teal`（UI操作用）を混同しないよう注意します。

## 実装ステップ

壊れにくくするため、次の順番で進めます。

### Step 1: スマホ用の土台 CSS

- `body` の `min-width: 1200px` をスマホ幅では解除
- `body` の `min-height: 760px` もスマホ幅では解除
- `.app-shell` をスマホでも `100vw / 100vh` に保つ
- `top-bar` をスマホ用に圧縮
- `top-bar` 高さを変える場合は、`.modal-backdrop` の `inset` も連動させる

### Step 2: ハンバーガー drawer

- `App.tsx` に `isMenuOpen` state を追加
- `lucide-react` から `Menu` アイコンを import する
- `top-bar` にハンバーガーボタンを追加
- スマホ幅で `.side-panel` を drawer 化
- overlay を追加
- 閉じるボタンを追加
- drawer と overlay の `z-index` を明示する
- `care-mode-control` を drawer に入れるため、JSX 側で配置を分ける
- ハンバーガーの `onClick` で `setIsMenuOpen(true)` と同時に `setModalToilet(null)` を呼び、開いていた詳細 bottom sheet を閉じる
- スマホ幅かつ drawer が閉じている間は、`aside.side-panel` に `aria-hidden` / `inert` を付けてフォーカス・読み上げから外す

### Step 3: スマホでの候補選択挙動

- `matchMedia` などで JS 側のスマホ判定を追加する
- 候補カードタップ時に drawer を閉じる
- 選択トイレへ `flyTo`
- スマホでは詳細 bottom sheet も開く
- 地図ピンタップ時も、スマホでは1回で詳細 bottom sheet を開く
- `handleToiletSelect` と `MapView` 内の `handleMapPinClick` の両方を更新する
- `MapView` のマーカー再生成 `useEffect` の依存配列に `isMobileLayout` を含める（768px 境界をまたいだ後も最新の判定でクリックハンドラを張り直すため）

### Step 4: 詳細 bottom sheet

- `ToiletModal` をスマホ幅では bottom sheet 表示にする
- PC 版の中央モーダル挙動は維持
- 写真の高さや余白をスマホ用に調整
- `.toilet-modal` の PC 用 `width: min(440px, calc(100vw - 720px))` をスマホメディアクエリで必ず上書きする
- base の `overflow: visible` を引きずらないよう、スマホメディアクエリで `overflow-y: auto` を明示する

### Step 5: 確認と微調整

- 375x667 程度の小型スマホ viewport
- 390x844 程度のスマホ viewport
- 430x932 程度の大きめスマホ viewport
- 768px 幅の境界
- PC 版 1920x1080

## 確認観点

- スマホで横スクロールが発生しない
- ハンバーガーが押しやすい
- drawer が地図の上に正しく表示される
- overlay タップで閉じられる
- 検索後に地図が移動する
- 候補選択後に drawer が閉じる
- 詳細カードが下から出る
- 詳細カードが画面外にはみ出さない
- PC 版の左パネル表示が壊れていない
- 詳細 bottom sheet を表示中にハンバーガーをタップしても、bottom sheet と drawer が重ならない（bottom sheet が閉じて drawer だけが開く）
- drawer が閉じている間、Tab キーで中の検索 input やケアモードボタンにフォーカスが移らない
- 写真や理由タグが多いトイレの詳細を開いても、bottom sheet の中身が `max-height` を超えた分はスクロールでき、画面外にはみ出さない
- 768px 境界をまたいでリサイズした直後でも、地図ピンタップの1タップ挙動（開くかどうか）が新しい画面幅の判定に従う
- `npm run build` が通る

## 初期リリースでやらないこと

最初のスマホ対応では、次は後回しにします。

- bottom sheet のスワイプ開閉
- 詳細カードの複数段階展開
- ピンのクラスタリング
- モバイル専用の経路検索
- iOS / Android それぞれの細かいジェスチャー最適化

まずは「スマホで目的地を入力し、地図を操作し、候補を選び、詳細を確認できる」状態を目標にします。
